# Flathub Publishing Guide

This document covers the complete workflow for building, testing, publishing, and updating
the app on Flathub.

---

## Background: why source builds?

Flathub requires **full offline source builds** ‚Äî the sandbox has no network access during
`flatpak-builder`. Every dependency (Flutter SDK, Dart packages, C libraries) must be declared
as a source with a verified SHA-256 checksum.

GitHub CI uses the same approach (`build-flatpak.yml`) so CI-produced artifacts and Flathub
builds are identical.

## How the offline build works

`flatpak-builder` fetches all sources and verifies SHA-256 checksums before any build
command runs. Two modules build in sequence.

### Module 1: `bclibc` (cmake-ninja)

Builds `libbclibc_ffi.so` from source and installs it to `/app/lib64/`.

```
source: git @ v1.0.5
builds: libbclibc_core.a + libbclibc_ffi.so
installs: /app/lib64/libbclibc_ffi.so, /app/include/bclibc/...
```

### Module 2: `ebalistyka` (simple)

Sources applied in order:

```
1. App source          git @ <tag>            ‚Üí /run/build/ebalistyka/
2. Flutter SDK         git @ 3.44.0           ‚Üí /run/build/ebalistyka/flutter/
   + engine artifacts  ~17 archives           ‚Üí flutter/bin/cache/artifacts/...
   + shared.sh.patch                          ‚Üí patches pub upgrade ‚Üí pub get --offline
   + setup-flutter.sh script
3. objectbox-c         archive x64/aarch64   ‚Üí /run/build/ebalistyka/objectbox-c/
4. generated-sources.json                    ‚Üí pub packages into .pub-cache/
   (archive + inline per package)
5. objectbox patch     (after pub sources!)   ‚Üí patches objectbox CMakeLists.txt
```

Build commands:
```
7 stamp copies ‚Üí setup-flutter.sh ‚Üí flutter build linux --release --no-pub ‚Üí install
```

### What `generated-sources.json` contains

The file is produced by `flatpak_gen sources` and contains three groups of entries:

**Pub packages** ‚Äî two entries per package from `pubspec.lock` and `flutter_tools/pubspec.lock`:
1. `type: archive` ‚Äî package tarball into `.pub-cache/hosted/pub.dev/$name-$version/`
2. `type: inline` ‚Äî SHA-256 hash into `.pub-cache/hosted-hashes/pub.dev/$name-$version.sha256`

Both are required. `pub get --offline` checks for the hash file and fails without it.

`flutter_tools/pubspec.lock` must be included because Flutter bootstraps itself by running
`pub get` inside `flutter/packages/flutter_tools/` before any `flutter` command. When the
app and flutter_tools need different versions of the same package (e.g. `json_annotation
4.11.0` vs `4.9.0`), both versions are included ‚Äî deduplication is on `(name, version)`
pairs, not just name.

**Flutter SDK artifacts** ‚Äî git source, engine archives, fonts, Gradle wrapper, stamps.
Also includes:
- `type: inline` for `flutter/bin/cache/pkg/sky_engine/pubspec.yaml` ‚Äî `packages/sky_engine/`
  was removed from the Flutter git tree in Flutter 3.x; the inline source guarantees the file
  is present with the correct null-safety environment constraint for `pub get --offline`.
- `type: patch` for `shared.sh.patch` ‚Äî see below.
- `type: script` for `setup-flutter.sh` ‚Äî runs `flutter pub get --offline`.

**Patch and script** are auto-generated by `flatpak_gen` and written to
`flatpak/patches/flutter/shared.sh.patch` alongside the JSON file.

### Why the shared.sh patch

Flutter's `flutter/bin/internal/shared.sh` bootstraps the Flutter tool by running
`pub upgrade --suppress-analytics` inside `flutter/packages/flutter_tools/`. `pub upgrade`
seeks newer versions and requires network access ‚Äî it fails in the sandbox.

The patch (built into `flatpak_gen`, written at generation time) replaces the command with
`pub get --offline --suppress-analytics`, which reads `pubspec.lock` directly and only needs
the exact pinned versions already in `.pub-cache/`.

### Why the objectbox patch

`objectbox_flutter_libs` uses CMake `FetchContent_Populate` to download `libobjectbox.so`
at build time. This deprecated API creates a cmake subbuild that doesn't inherit variables
from the parent build, so the standard `FETCHCONTENT_SOURCE_DIR_*` override doesn't work.

The patch adds `OBJECTBOX_PREBUILT_DIR` support ‚Äî the manifest passes
`-DOBJECTBOX_PREBUILT_DIR=/run/build/ebalistyka/objectbox-c` to `flutter build linux`,
pointing to the pre-seeded archive source. The objectbox patch `dest` must come **after**
`generated-sources.json` in the manifest sources list, otherwise the package directory
doesn't exist yet when the patch is applied.

## Regenerating sources

**CI regenerates sources automatically** on every build ‚Äî `generated-sources.json` is never
committed. For local flatpak-builder runs you must generate it first:

```bash
bash scripts/update-sources.sh
```

The script:
1. Clones Flutter at the version in `flatpak/flutter.version`
2. Clones and compiles `o-murphy/flutter_flatpak_gen`
3. Runs `flatpak_gen sources` with both lock files:
   - `pubspec.lock` ‚Äî app dependencies
   - `$FLUTTER_ROOT/packages/flutter_tools/pubspec.lock` ‚Äî Flutter tool dependencies
4. Writes `flatpak/generated-sources.json` and `flatpak/patches/flutter/shared.sh.patch`

`generated-sources.json` and `patches/flutter/shared.sh.patch` are **not committed** ‚Äî they
are regenerated on every CI run and locally before each local build.

Regenerate locally whenever `pubspec.lock` changes or Flutter is upgraded.

## Local testing

### Prerequisites (one-time setup)

```bash
sudo apt install flatpak flatpak-builder
flatpak remote-add --user --if-not-exists flathub https://flathub.org/repo/flathub.flatpakrepo
flatpak install --user flathub \
  org.freedesktop.Platform//25.08 \
  org.freedesktop.Sdk//25.08 \
  org.freedesktop.Sdk.Extension.llvm20//25.08
```

### Build and install

```bash
cd /path/to/ebalistyka

flatpak-builder --sandbox --user --install --install-deps-from=flathub --force-clean \
  --repo=.flatpak-repo --state-dir=.flatpak-builder \
  .flatpak-build flatpak/io.github.o_murphy.ebalistyka.yml
```

First run downloads all sources (~1‚Äì2 GB) into `.flatpak-builder/downloads/` ‚Äî cached
for subsequent builds. `--force-clean` removes only the build dir, not the cache.

### Run the installed app

```bash
flatpak run io.github.o_murphy.ebalistyka
```

### Export a standalone `.flatpak` file

```bash
flatpak build-bundle .flatpak-repo \
  ebalistyka_linux_x86_64.flatpak \
  io.github.o_murphy.ebalistyka
```

## Version update workflow

### 1. Bump the version in `pubspec.yaml`

```yaml
version: 0.1.15+42
```

### 2. Regenerate sources (if `pubspec.lock` changed)

```bash
bash scripts/update-sources.sh
git add flatpak/generated-sources.json
```

### 3. Update metainfo

Edit `flatpak/io.github.o_murphy.ebalistyka.metainfo.xml` ‚Äî add the new release:

```xml
<releases>
  <release version="0.1.15" date="2026-05-18"/>
  <release version="0.1.14" date="2026-05-14"/>
</releases>
```

### 4. Commit and push, then tag

```bash
git commit -m "chore: release v0.1.15"
git tag v0.1.15
git push origin main v0.1.15
```

The tag push triggers `release.yml` automatically, which builds `.flatpak` artifacts and
creates the GitHub release. The manifest's `tag`/`commit` fields are patched automatically
by `build-flatpak.yml` from the current git ref ‚Äî no manual editing needed.

### 5. Publish to Flathub

After the GitHub release is created, trigger manually:
**GitHub ‚Üí Actions ‚Üí Publish Flathub ‚Üí Run workflow ‚Üí tag: `v0.1.15`**

This runs `scripts/update-flathub.sh` which updates the Flathub repo and pushes ‚Äî
Flathub CI triggers automatically.

## Flutter version upgrade

When bumping Flutter (e.g. `3.44.0` ‚Üí `3.42.0`):

### 1. Update `flatpak/flutter.version`

```bash
echo "3.42.0" > flatpak/flutter.version
```

### 2. Regenerate sources

The Flutter version change means new SDK artifact URLs, new engine SHA-256 values,
and potentially new/changed flutter_tools dependencies ‚Äî all are handled by the same script:

```bash
bash scripts/update-sources.sh
git add flatpak/generated-sources.json flatpak/flutter.version
git commit -m "chore: upgrade Flutter to 3.42.0"
```

### 3. Verify the patch still applies

If Flutter changed `bin/internal/shared.sh`:

```bash
git clone --depth 1 --branch 3.42.0 https://github.com/flutter/flutter.git /tmp/flutter-new
patch --dry-run -p1 -d /tmp/flutter-new \
  < flatpak/patches/flutter/shared.sh.patch
```

If it fails, update the patch against the new file.

## ObjectBox version upgrade

When `objectbox_flutter_libs` is upgraded (e.g. `5.3.1` ‚Üí `5.4.0`):

### 1. Find new archive SHA-256

```bash
VERSION=5.4.0
curl -sL "https://github.com/objectbox/objectbox-c/releases/download/v${VERSION}/objectbox-linux-x64.tar.gz" | sha256sum
curl -sL "https://github.com/objectbox/objectbox-c/releases/download/v${VERSION}/objectbox-linux-aarch64.tar.gz" | sha256sum
```

### 2. Update the manifest

In `flatpak/io.github.o_murphy.ebalistyka.yml`, update the objectbox archive sources
and the patch `dest`:

```yaml
- type: archive
  only-arches: [x86_64]
  url: .../objectbox-linux-x64.tar.gz        # ‚Üê new version URL
  sha256: <new-sha256-x64>
  dest: objectbox-c
  strip-components: 0
- type: archive
  only-arches: [aarch64]
  url: .../objectbox-linux-aarch64.tar.gz    # ‚Üê new version URL
  sha256: <new-sha256-aarch64>
  dest: objectbox-c
  strip-components: 0
```

```yaml
- type: patch
  dest: .pub-cache/hosted/pub.dev/objectbox_flutter_libs-5.4.0   # ‚Üê new version
  path: patches/objectbox_flutter_libs/CMakeLists.txt.patch
```

### 3. Verify the patch still applies

```bash
patch --dry-run -p1 \
  -d ~/.pub-cache/hosted/pub.dev/objectbox_flutter_libs-5.4.0 \
  < flatpak/patches/objectbox_flutter_libs/CMakeLists.txt.patch
```

If it fails, regenerate the patch:

```bash
# Make the same OBJECTBOX_PREBUILT_DIR change manually, then:
diff -u \
  ~/.pub-cache/hosted/pub.dev/objectbox_flutter_libs-5.4.0/linux/CMakeLists.txt \
  /tmp/CMakeLists.txt.patched \
  > flatpak/patches/objectbox_flutter_libs/CMakeLists.txt.patch
```

## Publishing to Flathub (first submission)

The Flathub submission needs a per-app repo at
`https://github.com/flathub/io.github.o_murphy.ebalistyka`. The repo must contain:

```
io.github.o_murphy.ebalistyka.yml   ‚Üê the manifest
generated-sources.json              ‚Üê pub + SDK sources
patches/                            ‚Üê flutter/ and objectbox_flutter_libs/
```

Desktop, metainfo, icon, and wrapper script are fetched from the app's git source at
build time ‚Äî no copy needed in the Flathub repo.

Validate before submitting:

```bash
flatpak run --command=appstreamcli org.freedesktop.Sdk//25.08 \
  validate flatpak/io.github.o_murphy.ebalistyka.metainfo.xml
```

## Troubleshooting

### `pub get --offline` fails: "X doesn't match any versions"

A package required during the build is not in `.pub-cache`. Most likely cause:

- **flutter_tools dependency missing**: ensure `scripts/update-sources.sh` was run
  with a Flutter clone that matches `flatpak/flutter.version`. The script includes
  `$FLUTTER_ROOT/packages/flutter_tools/pubspec.lock` automatically.
- **Version conflict between lock files**: if the app and flutter_tools need different
  versions of the same package, both must be in the cache. `flatpak_gen` handles this
  by deduplicating on `(name, version)` pairs ‚Äî make sure you are running the latest
  version of the tool.

Regenerate sources and rebuild:

```bash
bash scripts/update-sources.sh
git add flatpak/generated-sources.json && git commit -m "chore: update flatpak sources"
```

### Patch fails: "can't find file to patch"

The objectbox patch `dest` path in the manifest doesn't match the installed version.
Check the actual version:

```bash
grep -A3 'objectbox_flutter_libs:' pubspec.lock
```

Update the `dest` in the manifest to match.

The patch must also be listed **after** `generated-sources.json` in the manifest sources
list ‚Äî the package directory must exist before the patch is applied.

### Build fails with "network access" errors

Something is trying to download at build time. Common causes:

- **Missing pub package**: a package in `pubspec.lock` is absent from
  `generated-sources.json`. Regenerate sources.
- **objectbox download**: the CMakeLists.txt patch didn't apply. Check source ordering
  in the manifest and verify the patch applies cleanly (see above).

### Flathub CI passes locally but fails on Flathub

- Flathub builds `x86_64` and `aarch64`. Test aarch64 locally with `--arch=aarch64`
  if you have the runtime installed.
- Flathub uses a stricter sandbox. Always build locally with `--sandbox` (already in
  the test command above).
